# 재화·포인트 시스템 기획서

> FC 리그 참가자들이 리그 활동을 통해 획득한 사이트 내 전용 포인트를 사용하여,
> 실제 축구 경기 예측 및 사이트 내 콘텐츠를 이용할 수 있는 포인트 기반 미니게임 시스템.
> **금전적 가치가 없는 커뮤니티 콘텐츠**다.

---

## 1. 포인트 기본 정책

### 1-1. 포인트의 성격

- 사이트 내부에서만 사용되는 **비현금성 재화**
- 현금 충전 불가 / 현금 환전 불가 / 상품 교환 불가 / 기프티콘 교환 불가 / 외부 플랫폼 이전 불가
- 오직 사이트 내 콘텐츠 이용, 랭킹, 기록, 명예 요소를 위해 사용

### 1-2. 핵심 설계 원칙

1. 포인트는 현금성 가치가 없는 사이트 내부 재화다
2. 모든 포인트 변동은 **원장(point_transactions)**에 기록한다
3. 유저 잔액(`user_wallet.balance`)은 원장 합계와 항상 일치해야 한다
4. 모든 포인트 획득·소모 내역은 사이트에서 **공개**한다
5. 포인트 전체 초기화는 여러 시즌 후 운영자가 수동으로 결정한다
6. 초기화 당시 누적 자산에 비례한 별도 보상을 지급한다
7. 승부 예측은 참가자 수가 적으므로 외부 배당 기반 **고정 배당** 방식을 사용한다
8. 재화 밸런싱 기준은 절대 가격이 아닌 **평균 시즌 수입 대비 비율**로 잡는다
9. 첫 1~2시즌은 **테스트 시즌**으로 운영하며, 데이터 기반으로 가격·보상을 조정한다

## 2. 포인트 획득 구조

| 획득처 | 설명 |
|--------|------|
| FC 리그 경기 결과 보상 | 승리·무승부·패배에 따른 포인트 지급 |
| 시즌 종료 순위 보상 | 시즌 종료 시 순위별 차등 지급 |
| 이벤트 보상 | 운영자가 기획한 이벤트 |
| 운영자 지급 | 관리자 직접 지급 |
| 승부 예측 적중 보상 | 예측 성공 시 배당 기반 지급 |
| 기타 사이트 활동 보상 | 추후 확장 |

### 시즌 종료 순위 보상 예시

```
1위: +1000P
2위:  +700P
3위:  +500P
4위:  +300P
참가 보상: +100P
```

## 3. 포인트 소모 구조

| 소모처 | 역할 | 특징 |
|--------|------|------|
| 승부 예측 참여 | 포인트 증식형 | 가장 큰 변동성, 실패 시 소모 |
| 선수 뽑기 | 중간 규모 소모 | 무작위 보상, 반복 사용 |
| 선수 강화 | 핵심 소각 콘텐츠 | 성공/실패 구조, 인플레이션 제어 |
| 특성 구매 | 장기 투자형 | 높은 가격, 보유 가치 높음 |
| 소모 아이템 구매 | 소규모 반복 소모 | 낮은 가격, 자잘한 회수 장치 |

## 4. 승부 예측 시스템

### 4-1. 예측 대상

운영자가 지정한 실제 축구 경기 (월드컵, 챔피언스리그, 국가대표, FC 이벤트 경기 등)

### 4-2. 배당 방식

외부 스포츠 배당 API 참고 **고정 배당** 방식.
(패리뮤추얼 방식 미사용 — 참가자 수가 적어 변동 배당이 부적합)

### 4-3. 예측 흐름

```
운영자 경기 등록 + 배당 설정
    ↓
참가자 포인트 배팅 (경기 시작 전까지)
    ↓
마감 시 배당 고정 (이후 변경 불가)
    ↓
경기 종료 후 결과 동기화 (자동 우선, 수동 보조)
    ↓
정산 실행 — 적중자에게 지급
```

### 4-4. 정산 공식

```
지급 포인트 = 참여 포인트 × 확정 배당
```

예시:
```
참여 포인트: 100P
확정 배당: 2.10
적중 시 지급: 210P  (100P × 2.10)
실패 시: 100P 소멸
```

### 4-5. 중복 정산 방지 (필수)

- 이미 정산 완료된 예측은 재정산하지 않는다
- 정산 트랜잭션은 고유 ID를 가진다
- 동일 경기·유저·예측에 대해 보상이 중복 지급되지 않아야 한다
- 수동 정산 버튼을 여러 번 눌러도 결과는 한 번만 반영된다

## 5. 포인트 원장 시스템

### 5-1. 원칙

현재 포인트를 단순 수정하는 방식이 아닌, **모든 증가·감소를 거래 내역으로 남기는 원장 방식** 사용.

> 유저 잔액을 직접 UPDATE 하는 코드는 절대 작성하지 않는다.

### 5-2. point_transactions 테이블 필드

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `transaction_id` | UUID | 거래 고유 ID |
| `user_id` | UUID | 대상 유저 |
| `amount` | INT | 증감량 (획득 양수, 소모 음수) |
| `transaction_type` | VARCHAR | 거래 유형 (아래 목록 참조) |
| `source_type` | VARCHAR | 거래 발생 원인 |
| `source_id` | UUID/INT | 관련 경기·예측·아이템 ID |
| `description` | TEXT | 사용자에게 표시할 설명 |
| `balance_after` | INT | 거래 반영 후 잔액 |
| `created_at` | TIMESTAMPTZ | 거래 생성 시각 |
| `created_by` | UUID/VARCHAR | 시스템 또는 운영자 ID |

### 5-3. 거래 유형 (transaction_type)

```
LEAGUE_MATCH_REWARD   — 리그 경기 결과 보상
SEASON_RANK_REWARD    — 시즌 종료 순위 보상
EVENT_REWARD          — 이벤트 보상
ADMIN_GRANT           — 운영자 지급
PREDICTION_STAKE      — 승부 예측 참여 (차감)
PREDICTION_REWARD     — 승부 예측 적중 보상
GACHA_PURCHASE        — 선수 뽑기
ENHANCEMENT_COST      — 선수 강화
TRAIT_PURCHASE        — 특성 구매
ITEM_PURCHASE         — 소모 아이템 구매
ADMIN_DEDUCT          — 운영자 회수
RESET_DEDUCT          — 포인트 초기화
CORRECTION            — 오류 보정
```

### 5-4. user_wallet 캐시

- 성능을 위해 `user_wallet` 테이블에 현재 잔액을 캐싱한다
- `user_wallet.balance`는 원장 합계와 **항상 일치**해야 한다
- 불일치 발생 시 원장 합계를 기준으로 복구한다

## 6. 포인트 공개 히스토리

### 6-1. 공개 범위

모든 참가자는 **모든 유저의** 포인트 획득·소모 내역을 확인할 수 있다.

### 6-2. 표시 정보

```
유저명 / 증감 포인트 / 거래 유형 / 거래 설명 / 관련 콘텐츠 / 거래 시각 / 거래 후 잔액
```

예시:
```
김철수  +500P   FC 리그 승리 보상        2026-06-18 21:30   잔액: 3,200P
김철수  -100P   한국 vs 일본 승부예측 참여  2026-06-18 22:10   잔액: 3,100P
김철수  +210P   한국 승 적중 보상         2026-06-19 01:30   잔액: 3,310P
```

### 6-3. 목적

운영 투명성 확보 / 유저 간 신뢰 확보 / 오류 추적 / 정산 검증 / 경쟁 콘텐츠 강화

## 7. 재화 밸런싱 기준

### 7-1. 핵심 기준값

```
평균 유저가 한 시즌 동안 획득하는 포인트
```

이를 기준 단위로 삼아 상품 가격을 비율로 설계한다.

### 7-2. 권장 가격 비율 (초기 테스트 시즌 기준)

| 소모처 | 평균 시즌 수입 대비 비율 |
|--------|------------------------|
| 승부 예측 1회 참여 | 5 ~ 10% |
| 소모 아이템 | 10 ~ 20% |
| 선수 강화 1회 | 20 ~ 50% |
| 선수 뽑기 1회 | 30 ~ 70% |
| 특성 구매 | 50 ~ 100% |

### 7-3. 예시 (평균 시즌 수입 1500P 가정)

| 소모처 | 포인트 |
|--------|--------|
| 승부 예측 1회 | 100P |
| 소모 아이템 | 200P |
| 선수 강화 1회 | 500P |
| 선수 뽑기 1회 | 700P |
| 특성 구매 | 1000P |

> 현재 구현된 가격: 소모 아이템 60P, 감독 특성 120P (shopProducts.js)
> 위 예시는 밸런싱 정책이 확정되면 갱신 필요

## 8. 테스트 시즌 운영 정책

### 8-1. 목적

첫 1~2시즌의 목적은 **완벽한 밸런스가 아닌 데이터 수집**이다.

수집 항목: 유저별 시즌 획득·소모 포인트 / 전체 생성·소모·유통 포인트 / 예측 참여 빈도·금액 / 뽑기·강화·특성 이용 패턴

### 8-2. 조정 방식 예시

```
시즌 1: 포인트 지급량 높게, 상품 가격 낮게 → 이용 패턴 확인
시즌 2: 획득량 일부 감소, 강화 비용 증가, 특성 가격 증가, 예측 최소 참여 포인트 조정
```

### 8-3. 테스트 시즌 이후

- 포인트 즉시 초기화 없음
- 3시즌부터 밸런스 조정 예고 공지 필수

## 9. 장기 포인트 누적 및 초기화

- 포인트는 매 시즌 종료 시 초기화하지 않는다
- 여러 시즌에 걸쳐 계속 누적된다
- 운영자가 **약 5~6시즌 후** 적절한 시점에 전체 초기화를 수동으로 결정

### 9-1. 초기화 판단 기준

- 포인트 격차가 과도하게 벌어진 경우
- 전체 유통 포인트가 지나치게 증가한 경우
- 신규 유저 진입 장벽이 커진 경우
- 장기 누적 랭킹 보상이 필요한 시점
- 시스템 개편이 필요한 시점

### 9-2. 초기화 당시 보상

누적 자산에 비례한 별도 보상 제공 (비현금).

예시: 누적 포인트 랭킹 칭호 / 명예 업적 / 특별 뱃지 / 다음 시즌 시작 보너스

## 11. 개발 시 필수 주의사항

### 포인트 변경은 반드시 원장으로만

```js
// ❌ 절대 금지
await db.query('UPDATE user_wallet SET balance = balance + $1 WHERE user_id = $2', [amount, userId]);

// ✅ 올바른 방식
await createPointTransaction({ userId, amount, type, sourceType, sourceId, description });
```

### 중복 처리 방지

시즌 보상, 예측 정산, 뽑기, 강화, 특성 구매, 초기화 등 모든 중요 작업은 **멱등성** 보장.

### 잔액 부족 검증

포인트 소모 전 반드시 잔액 확인. 부족 시 거래 생성 없이 실패 처리.

### 트랜잭션 처리

포인트 차감 + 콘텐츠 지급은 하나의 DB 트랜잭션으로 묶는다. 중간 실패 시 전체 롤백.

```
뽑기 예시:
1. 잔액 확인
2. 포인트 차감 거래 생성
3. 선수 지급 기록 생성
4. user_wallet 잔액 갱신
→ 하나라도 실패 시 전체 롤백
```

## 12. 시스템 핵심 흐름

```
FC 리그 참여
    ↓
리그 결과·시즌 순위에 따라 포인트 획득
    ↓
포인트로 승부예측·뽑기·강화·특성·아이템 구매
    ↓
모든 포인트 변동 원장 기록
    ↓
모든 유저가 히스토리 조회 가능
    ↓
여러 시즌 동안 포인트 누적
    ↓
운영자가 필요 시 전체 초기화
    ↓
초기화 당시 누적 자산 기준으로 별도 보상
```
